# 重构期间分支协作规则（五域收敛）

## 目的

在五域收敛重构进行期间，保证两件事同时成立：

- `dev` 分支可以继续承接业务开发
- 重构分支不会因为基线持续漂移而失去可合并性

本规则适用于以下分支：

- `master`
- `dev`
- `test`
- `codex-five-domain-dev`

## 分支职责

### `master`

- 作为主线稳定分支
- 当前与 `test` 指向同一提交时，不作为新增功能开发分支
- 不直接承接五域重构过程中的中间态提交

### `dev`

- 作为业务功能持续开发主线
- 允许继续提交新功能、缺陷修复、页面调整、测试补充
- 不允许继续扩大现有架构耦合

### `test`

- 用于测试验证与集成联调
- 当前若与 `master` 同步，则不单独承担开发基线职责

### `codex-five-domain-dev`

- 作为五域收敛重构专用分支
- 基线固定来源于 `origin/dev`
- 所有工程骨架、共享层拆分、五域迁移、依赖治理均在此分支完成

## 核心原则

### 原则 1：业务开发不停，但架构继续收口

不冻结 `dev` 的常规业务开发。  
冻结的是会继续放大耦合和边界混乱的改动。

### 原则 2：重构分支必须持续同步 `dev`

`codex-five-domain-dev` 不能长时间脱离 `dev`。  
必须定期把 `dev` 的最新变更并入重构分支。

### 原则 3：先约束边界，再允许新增能力落位

新增功能可以继续进入 `dev`，但不得通过新增跨服务直连、继续堆叠 `common`、新增散乱模块等方式落地。

### 原则 4：按阶段回合并，不做超长寿命大爆炸合并

五域重构不应等全部完成后一次性回合并。  
应按阶段、按域、按共享层逐步回合并。

## `dev` 分支允许的改动

以下改动允许继续进入 `dev`：

- 单模块内部业务逻辑增强
- Controller / Service / Mapper 内部功能补充
- SQL、索引、查询优化
- 页面字段增删、交互调整、样式优化
- API 测试、单元测试、集成测试补充
- Bug 修复
- 不改变服务边界的 DTO / VO 调整

## `dev` 分支限制的改动

以下改动默认禁止直接进入 `dev`，除非先做架构评审：

- 新增微服务或新增并列模块
- 新增跨服务直接依赖
- 一个服务直接引用另一个服务的 `entity`、`mapper`、`service`、`controller`
- 继续向 `data-platform-common` 或 `data-platform-api` 塞入业务实体、业务逻辑、跨域实现
- 大范围包结构重组
- 修改服务命名规范、应用名规范、网关聚合边界
- 新增全包扫描
- 在 `gateway` 中依赖业务服务实现模块

## 同步节奏

### 最低同步要求

`codex-five-domain-dev` 至少满足以下之一：

- 每天同步一次 `dev`
- 每完成一个重构阶段同步一次 `dev`
- 在 `dev` 出现共享层或核心调用链变更后立即同步

### 不允许的做法

- 连续多天不同步 `dev`
- 等重构临近完成时一次性吞入全部 `dev` 差异

## 推荐同步流程

1. 在 `dev` 上确认新增提交范围
2. 将 `dev` 合并到 `codex-five-domain-dev`
3. 优先处理共享层冲突
4. 再处理领域边界冲突
5. 最后执行编译和测试验证

## 冲突处理优先级

发生冲突时按以下顺序处理：

### 第一优先级：保留 `dev` 上的有效业务功能

如果冲突涉及最新业务逻辑、SQL 修复、接口字段补充、页面能力新增，默认保留 `dev` 语义。

### 第二优先级：保留重构分支上的边界治理结果

如果冲突涉及：

- 模块聚合结构
- 共享模块拆分
- 五域目录布局
- 依赖方向治理
- Feign 契约边界

默认保留重构方案。

### 第三优先级：冲突后统一重写，不做机械拼接

如果两边都合理，但实现方式不同，不直接拼接代码。  
应在新的目标结构下重写实现，避免把旧耦合重新带回重构分支。

## 回合并策略

建议按以下顺序回合并：

1. 根 `pom` 与共享层拆分
2. `masterdata + access + billing`
3. `identity`
4. `governance`
5. `sdk / test / web` 适配
6. legacy 模块删除

每次回合并前应满足：

- `mvn -q -DskipTests compile` 通过
- 关键测试通过
- 新旧依赖方向符合目标规则

## 架构守门规则

在重构完成前，所有涉及以下内容的 `dev` 改动必须先审：

- 新增对其他模块源码包的直接引用
- 新增 `common` 里的业务实现
- 新增跨服务写操作
- 新增共享实体
- 新增网关到旧模块的专属路由

如果确实需要新增功能，优先采用以下路径：

1. 放在既有模块内部实现
2. 通过契约接口扩展
3. 在重构分支中直接落到目标五域结构

## 当前执行建议

当前仓库建议执行方式如下：

1. 所有常规业务功能继续进入 `dev`
2. 五域重构只在 `codex-five-domain-dev` 推进
3. 每天同步一次 `dev` 到重构分支
4. 每完成一个阶段立即做编译验证
5. 禁止在 `dev` 上继续做会扩大耦合的结构性改动

## 结论

当前不建议停止新功能开发。  
当前也不允许 `dev` 无约束继续发散。

正确做法是：

- `dev` 持续承接业务功能
- 重构分支持续吸收 `dev`
- 对结构性改动设置边界
- 按阶段回合并，而不是最终一次性合并
