# 模板化计费方案实现说明

## 1. 目标与边界

本实现把“如何计算”和“对谁、何时、依据什么事实计算”拆开：

- `billing_template`：平台内置算法模板，包括按次、阶梯、包次、包周期、按返回数据量、按时长。
- `billing_plan`：厂商 + 接口 + 会计方向的版本化实例，保存价格、计量条件、响应字段绑定、生效区间和契约指纹。
- `billing_event`：不可变计费事实账本。正常用量、周期固定费和冲正均追加事件，不修改历史价格。
- `billing_usage_balance`：按方案、账期和累计范围维护用量，服务于阶梯与套餐超额计算。

旧 `billing_rule`、`/internal/billing/calculate` 和日聚合接口保留兼容；新调用链使用计量策略与事件账本。

## 2. 运行时链路

1. Access 在调用厂商前，通过 `vendorCode + interfaceCode + callTime` 解析唯一生效方案。
2. Billing 返回方案 ID、版本、策略哈希和有限字段选择器。
3. Access 调用厂商并完成响应归一化、响应契约校验。
4. Access 只提取策略要求的响应/请求字段，连同成功状态、缓存命中、耗时等元数据提交 Billing；完整厂商响应不跨域传输。
5. Billing 校验固定方案版本和策略哈希，在事务内获取请求幂等锁、判断是否收费、占用账期用量、计算金额、写事件并更新兼容日聚合。
6. 同一个 `requestId` 重试直接返回已存在事件，不会重复扣量或重复入账。

## 3. 计量条件 DSL

条件仅支持有限操作符，不执行脚本或 SpEL：

- 存在性：`EXISTS`、`NOT_EXISTS`、`EMPTY`、`NOT_EMPTY`
- 布尔：`TRUE`、`FALSE`
- 等值/集合：`EQ`、`NE`、`IN`、`NOT_IN`
- 比较：`GT`、`GTE`、`LT`、`LTE`

多个条件支持 `AND` / `OR`。字段缺失可配置为待复核、不计费、仍计费或报错。缓存命中可配置免费、同价或自定义单价。

数量可取固定值、响应数值、数组长度或调用耗时。响应字段必须存在于 Masterdata 接口响应契约，且字段 ID、路径、类型必须一致。

## 4. 定价语义

- 按次/按返回数据量：`unitPrice × quantity`。
- 按时长：先按毫秒、秒、分钟或小时及取整规则折算数量，再乘单价。
- 阶梯：按账期累计量累进计价，只计算本次用量跨越各区间的增量金额。
- 包次：周期固定费用生成独立 `RECURRING_FEE` 事件；用量超过包含量后按超额单价计算。
- 包周期：每个账期只生成一次固定费用事件，用量事件金额为零。
- SLA：超过阈值后按每 100ms 补偿率降低费用，最低为零。

所有金额统一保留 8 位小数，账本同时保存基础金额、调整金额、最终金额和完整定价快照。

## 5. 生命周期与安全约束

- 状态：`DRAFT → PUBLISHED/ACTIVE → EXPIRED`；响应契约变化标记 `NEEDS_REVIEW`。
- 已发布版本禁止原地修改，只能创建下一版本。
- 同一方案版本的生效区间不可倒置；同一厂商、接口、会计方向的不同方案不可重叠。
- 发布时重新校验模板、厂商接口归属、响应字段、阶梯连续性、币种、时区和价格边界。
- 事件固定 `planId + planVersion + policyHash`，避免配置发布与并发调用之间发生漂移。
- 契约漂移每天检查，也可由管理端手动触发。待复核期间继续使用已固定的旧策略，避免调用链突然中断，但管理端会明确告警。

## 6. 冲正与对账

冲正通过 `POST /billing/event/{id}/reverse` 追加唯一负向事件：

- 原事件保持不变；冲正事件关联 `original_event_id`。
- 金额和数量取原事件负值。
- 用量事件会原子恢复对应账期累计用量。
- 相同冲正请求幂等；同一原事件最多存在一条全额冲正。

事件统计按正负事件求净额，可分别使用 `VENDOR_PAYABLE` 与 `INTERNAL_CHARGEBACK` 维护厂商应付和内部核算方案。

## 7. 管理端流程

计费管理新增“计费方案”工作区：

1. 选择计费模板。
2. 绑定厂商、接口、计费方向、币种和时区。
3. 从接口响应契约选择收费字段并配置判断条件、缺失策略、缓存策略和数量字段。
4. 配置单价、阶梯、套餐、时长与超额价格。
5. 配置生效区间和 SLA，保存草稿、模拟、校验并发布。

同一页面提供方案版本、契约变更检查、周期费用补提、事件账本查询和冲正入口。

## 8. 数据库与发布

执行 `sql/migrations/V021__create_billing_plan_and_event_ledger.sql`。`sql/init.sql` 和 `sql/create_database.sql` 已引用该迁移。

迁移会把可绑定厂商接口的旧规则平滑生成 `LEGACY-*` 方案，旧数据和旧接口不删除。生产执行前应先备份，执行后核对：模板 6 条、旧规则迁移数量、重复生效区间、事件表索引和定时任务时区。

## 9. 回归不变量

- 失败调用、免费缓存和条件不满足时金额必须为零。
- 缺少关键证据不能静默按默认值收费。
- 数量不能为负；阶梯必须从零连续覆盖到无上限。
- 同一请求无论重试或并发多少次，只能生成一个事件并占用一次用量。
- 任意历史事件都能从策略版本、证据哈希、判断明细和价格快照解释。
- 冲正后净金额、净数量和累计用量恢复正确。
